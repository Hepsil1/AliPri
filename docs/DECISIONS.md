# 🧠 Архитектурные решения (ADR) — СчётДетей

## Формат
Каждое решение документируется в формате:
- **Контекст** — почему возник вопрос
- **Решение** — что выбрали
- **Альтернативы** — что рассматривали
- **Последствия** — к чему приводит решение

## ADR-001: SQLite вместо PostgreSQL/MySQL

**Дата**: 2026-04-09  
**Статус**: ✅ Принято

### Контекст
Нужна база данных для хранения ~50 спортсменов, их абонементов и посещений. Проект используется одним человеком локально.

### Решение
Используем **SQLite** через библиотеку `better-sqlite3`.

### Альтернативы
| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| PostgreSQL | Масштабируемость, надёжность | Нужна установка сервера, overkill |
| MySQL | Популярность | Нужна установка, сложная настройка |
| JSON файлы | Простота | Нет транзакций, медленные запросы |
| **SQLite** ✅ | Zero-config, один файл, быстро | Один пользователь одновременно |

### Последствия
- ✅ Бэкап = копирование одного файла `club.db`
- ✅ Никакой установки серверов БД
- ✅ Синхронный API (better-sqlite3) — проще код
- ⚠️ Если потребуется многопользовательский доступ — придётся мигрировать

## ADR-002: Vanilla JS вместо React/Vue

**Дата**: 2026-04-09  
**Статус**: ✅ Принято

### Контекст
Интерфейс относительно простой: 5 страниц, списки, формы, модалки. Один разработчик.

### Решение
Используем **Vanilla JavaScript** без фреймворков.

### Альтернативы
| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| React | Компонентность, экосистема | Сложность, build step, зависимости |
| Vue | Простота, реактивность | Лишняя абстракция для 5 страниц |
| **Vanilla JS** ✅ | Ноль зависимостей, быстрый старт | Больше кода для DOM-манипуляций |

### Последствия
- ✅ Ноль зависимостей на фронтенде
- ✅ Мгновенная загрузка страницы
- ✅ Легко отлаживать — всё прозрачно
- ⚠️ Придётся писать свой простой роутер

## ADR-003: SPA с клиентской маршрутизацией

**Дата**: 2026-04-09  
**Статус**: ✅ Принято

### Контекст
Нужна навигация между разделами (дашборд, группы, спортсмены, журнал).

### Решение
Одна HTML-страница, маршрутизация через hash-роутер (`#/groups`, `#/athletes`).

### Последствия
- ✅ Один файл index.html
- ✅ Мгновенная навигация без перезагрузки
- ✅ Простая реализация на Vanilla JS

## ADR-004: REST API с JSON

**Дата**: 2026-04-09  
**Статус**: ✅ Принято

### Контекст
Frontend и Backend на одном сервере, но разделены архитектурно.

### Решение
REST API эндпоинты возвращают JSON. Frontend общается через `fetch()`.

### Последствия
- ✅ Чистое разделение frontend/backend
- ✅ Легко тестировать API отдельно (curl, Postman)
- ✅ Можно в будущем подключить мобильное приложение

## ADR-005: Уведомления — текст для копирования (не автоотправка)

**Дата**: 2026-04-09  
**Статус**: 🟡 Ожидает подтверждения

### Контекст
Тренер отправляет голосовые в Telegram при истечении абонемента.

### Решение (предложение)
Показываем готовый **текст для копирования** в Telegram. Не делаем автоматическую отправку через бота.

### Альтернативы
| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Telegram Bot API | Полная автоматизация | Нужен бот, токен, настройка |
| **Текст + копирование** ✅ | Просто, быстро | Ручное действие |
| Email | Универсально | Не все читают |

### Последствия
- ✅ Не нужна интеграция с Telegram API
- ✅ Тренер контролирует текст перед отправкой
- ⚠️ Можно позже добавить Telegram бота как улучшение
